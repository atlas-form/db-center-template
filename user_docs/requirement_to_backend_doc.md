# 用户需求到服务端实现文档

这个文档的作用，是把“用户说的业务需求”翻译成“AI 可以稳定开发的服务端实现方案”。

它不是代码文档，而是开发前的确认文档。

在本模板里，AI 开始写代码前，应该先产出这份文档给用户确认。

---

## 为什么需要这一步

很多时候用户说的是业务语言，例如：

- 我要一个订单系统
- 用户可以提交申请
- 管理员可以审核

但 AI 真正开发时需要的是：

- 有哪些表
- 字段是什么
- 哪些字段需要唯一
- 哪些数据要按用户隔离
- 接口长什么样
- 哪些接口需要登录

如果不先把这些内容整理清楚，AI 就容易边猜边写，最后反复返工。

所以推荐固定流程：

**用户需求 -> 服务端实现文档 -> 用户确认 -> 正式开发**

---

## 这份文档应该包含什么

一份合格的“服务端实现文档”至少应包含下面这些内容。

### 1. 业务理解

AI 先用自己的话复述用户需求，确认理解没有偏差。

例如：

- 这个系统要解决什么问题
- 核心业务对象是什么
- 用户要完成哪些操作

### 2. 数据表设计

列出准备新增或修改的表。

每张表应说明：

- 表名
- 表作用
- 是否与当前登录用户有关

### 3. 字段设计

每张表的每个字段至少要写清楚：

- 字段名
- 类型
- 是否必填
- 是否唯一
- 默认值
- 字段含义

### 4. 表关系

如果有多张表，需要说明：

- 一对一
- 一对多
- 多对多
- 哪张表负责持有外键

### 5. 权限与用户隔离

如果业务和登录用户有关，需要明确：

- 哪些数据属于当前用户
- 是否要保存 `user_id`
- 哪些接口只能访问自己的数据
- 是否有管理员接口

### 5.1 AI/LLM 调用设计

如果业务涉及 AI、LLM、图片识别、OCR、视觉理解或多模型切换，需要明确：

- 使用哪个 `config/services.toml` 里的 `llm.name`
- 输入是文本、图片 URL、base64 data URL，还是文件上传
- 输出是纯文本、结构化 JSON，还是需要落库
- 模型调用失败时接口如何返回错误
- 是否需要运行 `cargo run -p web-server --example llm_vision_smoke` 做验证

### 6. HTTP 接口设计

至少应列出：

- 请求方法
- 路径
- 功能说明
- 是否需要登录
- 查询的是自己的数据还是全部数据

### 7. 开发顺序

推荐明确写出：

1. migration
2. entity 生成
3. repo
4. service
5. web-server

## 推荐文档格式

下面是一份推荐模板。

```md
# 服务端实现文档

## 1. 业务理解

<AI 对业务的理解>

## 2. 数据表设计

### 2.1 表一：<table_name>

- 用途：<说明>
- 是否与当前登录用户相关：是 / 否

字段设计：

| 字段名 | 类型 | 约束 | 说明 |
| ------ | ---- | ---- | ---- |
| id | bigint | 主键 | 主键 |
| ... | ... | ... | ... |

### 2.2 表二：<table_name>

...

## 3. 表关系

<说明表之间的关系>

## 4. 权限设计

- 哪些接口需要登录
- 哪些接口只能访问自己的数据
- 哪些接口允许管理员访问

## 5. HTTP 接口设计

| 方法 | 路径 | 说明 | 是否登录 |
| ---- | ---- | ---- | -------- |
| GET | /xxx | 查询 | 是 |

## 6. AI/LLM 调用设计

如果不涉及 AI/LLM，写：不涉及。

如果涉及，写清：

- `llm.name`
- 输入格式
- 输出格式
- 错误处理
- 验证方式

## 7. 开发顺序

1. migration
2. entity
3. repo
4. service
5. web-server
```

## 示例

例如用户说：

```text
我要做一个任务系统。
用户可以创建任务、查看自己的任务。
管理员可以查看所有任务。
接口需要登录。
```

AI 先不要写代码，而是先输出类似这样的文档：

```md
# 服务端实现文档

## 1. 业务理解

该系统用于任务管理。
普通用户可以创建和查看自己的任务，管理员可以查看所有任务。

## 2. 数据表设计

### 2.1 task

- 用途：保存任务信息
- 是否与当前登录用户相关：是

字段设计：

| 字段名 | 类型 | 约束 | 说明 |
| ------ | ---- | ---- | ---- |
| task_id | bigint | 主键 | 任务主键 |
| owner_user_id | varchar(64) | 必填, 索引 | 当前登录用户 ID |
| title | varchar(255) | 必填 | 任务标题 |
| content | text | 可空 | 任务内容 |
| status | varchar(20) | 必填 | 任务状态 |
| created_at | timestamp | 必填 | 创建时间 |

## 3. 权限设计

- 创建任务需要登录
- 查询我的任务需要登录
- 管理员查询全部任务需要单独接口

## 4. HTTP 接口设计

| 方法 | 路径 | 说明 | 是否登录 |
| ---- | ---- | ---- | -------- |
| POST | /tasks | 创建任务 | 是 |
| GET | /tasks/my | 查询我的任务 | 是 |
| GET | /tasks | 管理员查询全部任务 | 是 |

## 5. 开发顺序

1. migration
2. entity
3. repo
4. service
5. web-server
```

等用户确认后，再进入编码阶段。

---

## 一句话总结

这份文档的目标不是“把代码写出来”，而是：

**先把业务需求翻译成可确认、可落地、可约束 AI 的服务端设计。**
